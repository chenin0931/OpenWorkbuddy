# Quiet Workbench design system

本文档定义 On My WorkBuddy Renderer 的视觉、交互和产品语言约束。实现来源以
`apps/desktop/src/renderer/src/design/`、`icons.tsx`、`ui.tsx` 和现有 Shell/Work
组件为准；本文不改变 Agent Host、权限、审批或持久化语义。

## 1. 设计原则

Quiet Workbench 是“安静的工作台”，不是聊天玩具或 Agent 调试控制台。

1. **工作内容优先。** 导航、状态和系统信息主动退后，用户请求、结果、变更和证据拥有最高层级。
2. **只在需要决定时打断。** 普通完成保持安静；确认、补充输入、暂停和失败才提升视觉权重。
3. **渐进披露。** 活动默认收起，原始参数、JSON、命令输出和诊断信息只在用户主动展开后出现。
4. **本机边界可感知。** 工作区、发送内容、影响范围、可逆性和检查结果必须用具体语言说明，不能依赖“安全”一词替代事实。
5. **少而一致。** 一屏只保留一个高权重主操作；卡片、阴影、强调色和徽章必须有明确的信息层级理由。
6. **成熟而不过度人格化。** 文案直接说明正在做什么、发生了什么和下一步是什么，不使用卖萌、魔法或拟人化承诺。

界面对象按“工作区 → 工作 → 用户请求”组织。计划、工具和检查属于一次请求的内部活动，不能与工作并列成一级产品对象。

## 2. 产品语言

### 2.1 导航与对象命名

- 一级导航使用“工作”和“自动化”。
- “资料库”承载记忆、连接和技能；“隐私与记录”归入“设置”。
- 用户创建的是“新工作”，历史区域称为“最近”。
- 文件修改称为“变更”，Artifact 面向用户称为“输出”，Tool Call 面向用户称为“活动”。
- Chrome Bridge 和 MCP 可以出现在设置、连接详情或诊断信息中；普通工作界面优先使用“浏览器连接”和“连接”。

### 2.2 用户可见状态

内部状态机不直接成为用户文案。`StatusBadge` 当前映射如下：

| 内部状态 | 用户文案 | 展示规则 |
| --- | --- | --- |
| `understanding`、`running` | 正在处理 | 仅当前工作需要进度反馈时显示 |
| `planning` | 正在整理 | 仅计划仍在生成时显示 |
| `verifying` | 正在检查 | 存在实际检查动作时显示 |
| `waiting_approval` | 需要确认 | 强提醒，并展示明确操作 |
| `waiting_user` | 等你回复 | 强提醒，并说明缺少的信息 |
| `paused` | 已暂停 | 显示继续入口 |
| `failed` | 未完成 | 说明影响与可执行的下一步 |
| `cancelled` | 已停止 | 说明已产生的变更是否保留 |
| `completed` | 不显示状态 | 结果本身即完成反馈 |

`verified` 和 `partial` 是结果证据，不是任务状态。使用“3 项检查通过”或“还有 1 项未检查”，不得渲染常驻的全局完成卡。

### 2.3 禁用术语和替代写法

以下限制适用于普通导航、对话和结果区域；设置、高级详情、日志导出等技术场景可以保留必要的协议名称。

| 避免 | 使用 |
| --- | --- |
| 任务、新建任务、最近任务 | 工作、新工作、最近 |
| Run、RunEvent、Checkpoint | 工作、活动记录、恢复点（仅确需解释时） |
| Tool Call、工具调用 | 活动、执行记录 |
| 运行上下文 | 本次使用 |
| 产物、Artifact | 输出、结果文件 |
| 连接器 | 连接 |
| 验证状态 | 检查结果 |
| 任务已完成、任务已部分完成 | 不显示；或“还有 X 项未处理” |
| Agent 执行失败 | 这次操作没有完成 |
| Harness、Schema、Provider | 权限与检查、参数格式、模型服务（普通界面） |

按钮使用具体动词，例如“运行检查”“允许这一次”“修改后允许”“拒绝”“撤销变更”。避免只有“确定”“执行”或“继续”的模糊按钮；高风险操作的按钮必须包含动作对象或风险含义。

## 3. 视觉令牌

所有功能样式优先使用 `tokens.css` 中的语义变量，不在业务组件中复制十六进制颜色、任意圆角或任意阴影。

### 3.1 语义颜色

| Token | 浅色 | 深色 | 用途 |
| --- | --- | --- | --- |
| `--canvas` | `#f4f5f7` | `#101216` | 应用底层画布 |
| `--surface` | `#ffffff` | `#171a1f` | 主内容和可交互表面 |
| `--surface-subtle` | `#f8f9fb` | `#1d2128` | 次级区域和检查器 |
| `--surface-muted` | `#eef0f3` | `#242932` | 控件底色、代码和弱分组 |
| `--sidebar` | `rgba(235, 237, 241, .94)` | `rgba(22, 25, 31, .96)` | 半透明侧栏 |
| `--text` | `#1b1d22` | `#f3f4f6` | 主文字 |
| `--text-secondary` | `#5c626c` | `#abb0b8` | 说明文字 |
| `--text-tertiary` | `#676d77` | `#9198a3` | 时间、元信息 |
| `--separator` | 10% 深色 | 10% 白色 | 普通分隔 |
| `--separator-strong` | 17% 深色 | 17% 白色 | 输入、浮层边界 |
| `--accent` | `#4057c7` | `#4164c7` | 主操作、选中和焦点 |
| `--success` | `#277a58` | `#5bc493` | 确认通过、健康、成功检查 |
| `--warning` | `#a76513` | `#e7a44c` | 需要关注或确认 |
| `--danger` | `#b6424a` | `#f07880` | 失败、破坏性或不可逆动作 |
| `--link` | `#315fbd` | `#91b4ff` | 链接 |

使用约束：

- Accent 只用于主操作、当前选择和焦点，不用于大面积装饰。
- Success 只表示真实成功信号；普通完成不使用绿色徽章。
- Warning 表示用户需要注意或决定，不能用于一般提示。
- Danger 只用于失败、删除和不可逆风险。
- 状态不能只靠颜色表达，必须同时有文字、图标或结构差异。
- 品牌节点色 `#a76513` 仅属于 `BrandMark`，不进入功能状态色体系。

### 3.2 字体与字号

字体栈为 macOS 系统字体、SF Pro Text、PingFang SC；代码和路径使用 `--font-mono`（SF Mono 优先）。

| 层级 | 规格 | 典型位置 |
| --- | --- | --- |
| Hero | `30px`，约 660 weight | 欢迎页主标题 |
| 页面标题 | `22px`，约 660 weight | 设置、资料库页面 |
| 工作标题 | `15px`，约 650 weight | 工作标题栏 |
| 对话正文 | `14px / 1.64` | 用户与 WorkBuddy 内容 |
| UI 正文 | `13px / 1.54` | 表单、按钮、说明 |
| 次级标签 | `12px` | 导航、活动、紧凑控件 |
| 元信息 | `11px` | 时间、路径、状态补充 |

用户可见文字不得低于 11px。层级依靠字号、字重、颜色和留白共同建立，不使用全大写或过多粗体制造层级。

### 3.3 间距、圆角、阴影与动效

- 间距只从 `--space-1/2/3/4/5/6/8/12` 选择，对应 `4/8/12/16/20/24/32/48px`。
- 控件使用 `--radius-control: 6px`；列表与内容表面使用 `--radius-surface: 8px`；Composer、审批和浮层使用 `--radius-floating: 12px`。
- `--shadow-floating` 只用于 Modal、Popover 和窄屏抽屉；`--shadow-composer` 只用于 Composer。普通列表、侧栏和静态内容不用阴影。
- 微交互使用 `--motion-fast: 120ms`，普通状态切换使用 `--motion-base: 180ms`，面板使用 `--motion-panel: 240ms`，缓动使用 `--ease-out`。

## 4. 图标与品牌资产

功能图标和品牌资产必须分离：

- 功能图标通过 `AppIcon` 或兼容别名 `Icon` 使用语义 `IconName`，业务组件不得直接导入其他 Phosphor 图标。
- 功能图标默认 `regular`；`fill` 只用于选中或需要注意的状态。常用尺寸为 14、16、18 和 24px，同一控件组保持一致。
- 只有图标的按钮必须使用 `IconButton` 并提供具体 `label`；图标本身保持装饰性，不重复朗读。
- `BrandMark` 是折叠工作面组成的 W 和暖色人工确认节点，只用于品牌、启动和产品身份区域。
- App/扩展图标、Onboarding 和大型空状态可以使用经过校正的品牌图形；位图或生成式素材不得充当 16–24px 功能图标。
- 不使用机器人、聊天气泡、魔杖、霓虹光晕或无意义 Sparkle 作为通用 AI 装饰。

## 5. 组件使用约束

以下约束对应当前已实现的 Renderer primitives，不另造组件 API。

- **Button：** `.button.primary` 在一个视图或决策区最多出现一次；次要操作使用 `.secondary` 或 `.ghost`；破坏性操作必须使用 danger 语义和明确动词。
- **IconButton：** `label` 必填；切换类按钮通过 `active` 暴露 `aria-pressed`，不能只靠图标颜色表示状态。
- **StatusBadge：** 只呈现有操作价值的非完成状态；`completed` 必须返回空。
- **Toggle：** 必须提供描述动作的 `label`，不能用相邻文字作为唯一可访问名称。
- **Modal / ConfirmDialog：** 统一使用现有组件，保留 Escape、焦点锁定、打开聚焦、关闭焦点恢复和遮罩关闭行为。高风险确认必须同时提供明确取消入口。
- **Tabs：** 使用 `Tabs` 生成 `tablist/tab/tabpanel`；每个 Tab 需要可读名称，并保留左右键及 Home/End 操作。
- **PanelResizer：** 使用 `PanelResizer` 和 `usePersistentPanelWidth`。右侧检查器默认 330px，限制 300–420px；必须保留 Pointer Capture、方向键、Shift 大步进和 separator ARIA。
- **Field：** `label` 必填；`hint` 用于格式或后果说明，错误信息不能只改变边框颜色。
- **EmptyState：** 说明当前为何为空，并最多提供一个主下一步；不得用多层卡片填满空白。
- **Toasts：** 用于非阻塞反馈，不承载必须由用户决定的审批或不可恢复错误。
- **活动与诊断：** 一次用户请求只呈现一个 WorkBuddy 响应块和一个默认收起的活动区域；原始 JSON、调用 ID 和日志放在二级诊断 Disclosure 中。
- **结果：** Diff、检查、来源和输出紧邻对应回答；检查器默认关闭，只显示实际有内容的“详细”“变更”“活动”。

侧栏保持浅层结构；主工作区是唯一视觉中心。静态内容优先使用留白和分隔线，只有审批、Composer、Modal 等具有明确边界或浮层关系的内容才使用强表面。

## 6. 主题与无障碍验收

### 6.1 浅色与深色

- 浅色使用 `:root` Token，深色由 `:root[data-theme="dark"]` 覆盖；组件不得通过条件 JSX 维护第二套颜色。
- 两种主题都必须检查主内容、侧栏、检查器、Composer、审批、Modal、代码块、链接、焦点和禁用状态。
- 正文与背景对比度至少 4.5:1；大号文字至少 3:1。分隔线不足以传达的信息必须由文字或结构补充。
- 深色不使用纯黑大底或发光紫色，浅色不使用大面积高饱和强调色。

### 6.2 高对比度

- `prefers-contrast: more` 会增强 separator 和次级文字；新增组件必须使用语义 Token 才能继承该行为。
- 在高对比模式检查所有焦点框、选中项、状态、边界和链接；不能出现仅靠低透明度区分的可交互元素。
- 所有状态必须在灰阶下仍可通过文字或图形识别。

### 6.3 Reduce Motion

- `prefers-reduced-motion: reduce` 会把动画和过渡压缩到近乎即时；新增动画必须允许该全局规则覆盖。
- 减少动画后仍要保留最终状态，不依赖位移、闪烁或循环动画解释进度。
- 面板、Disclosure、Spinner 和工作指示器都需在 Reduce Motion 下检查，不得阻挡操作或持续吸引注意。

### 6.4 键盘与辅助技术

- 所有交互必须有可见的 `:focus-visible`，不得移除 2px Accent 焦点框而不提供等价替代。
- Modal 必须可用 Escape 关闭并把焦点还给触发器；Tab/Shift+Tab 不能逃出打开的 Modal。
- Tabs 支持 Left/Right/Home/End；PanelResizer 支持 Left/Right、Shift 大步进以及 Home/End。
- 只有图标的控件、Switch、进度和 Separator 都必须有可访问名称或数值。
- 审批、失败和等待输入不能只用 Toast 或颜色通知；屏幕阅读器用户必须能定位正文和下一步操作。

## 7. 变更检查清单

提交 Renderer UI 变更前确认：

- 使用现有语义 Token、`IconName` 和 primitives，没有新增任意颜色、字号、圆角或阴影。
- 普通完成不显示状态卡；活动默认收起；同一请求不重复 WorkBuddy 身份。
- 文案使用“工作、活动、变更、输出、检查”，没有把内部运行概念暴露到普通界面。
- 主操作唯一且动词具体；危险操作说明目标、影响范围与可逆性。
- 浅色、深色、高对比度和 Reduce Motion 均完成目视检查。
- 键盘可完成导航、Tab 切换、Modal、审批和面板调整；焦点顺序与恢复正确。
- 运行 desktop typecheck、相关单元测试和对应 Playwright 场景，并检查固定视口下没有遮挡或溢出。
